# Windows 코딩 위임 경로 — 설계안 (조사·실측 기반, 구현 전 승인용)

작성: 백팀장 / 2026-07-07 / 스코프 C 후속
상태: **설계안(제안). 코딩 미착수.** 남수장님 승인 후 구현 라운드로 진행.
대상 독자: 남수장(결정권자), 자비서(PM)

---

## 0. 한 줄 요약

윈도우에서 codex 코딩 위임은 **"안전설정이면 0파일, 쓰기 되려면 bypass"라는 양립불가**가 근본
블로커다. 정공법은 **일회용 격리 사본에서만 bypass로 위임 → diff 리뷰 → 사람 승인 후 원본 apply**.
단 "bypass를 어느 범위에서 허용하느냐"가 핵심 결정 사항이라, 아래 3개 대안의 트레이드오프를 보고
남수장님이 범위를 정해주셔야 한다.

---

## 1. 근본 블로커 — 실측으로 확정 (감 아님)

리눅스 박스(codex-cli 0.142.3, ChatGPT auth, 임시 git repo)에서 MCP `codex` tools/call 로 직접 재현:

| sandbox 설정 | codex 응답 | 실제 파일 | 비고 |
|---|---|---|---|
| `workspace-write` + `never` (안전측) | `isError=false` + threadId 정상 | **0파일** | text 에 "sandbox/write failure". **응답은 성공처럼 보임** |
| `danger-full-access` | `isError=false` "Done." | **파일 2개(신규+수정)** | 쓰기 되는 유일 경로 |
| 읽기전용 workdir + danger-full-access | `isError=false` + threadId | **0파일** | text 에 "write operation was rejected" |

**핵심 3가지:**
1. **안전설정으론 아무것도 못 쓴다.** 리눅스는 bwrap 네임스페이스 생성 실패, 윈도우는 read-only 강등
   — *원인은 달라도 결과는 동일(0파일)*.
2. **쓰기 실패가 조용하다.** 0파일인데도 `isError=false`, exit 0. diff/파일존재로 확인 안 하면 못 잡는다.
   (→ 이래서 doctor 3번을 "응답 성공"이 아니라 "실제 파일 쓰기 성공"으로 판정하도록 바꿨다. 작업 A.)
3. **쓰기가 되는 유일 경로는 bypass(danger-full-access)** 뿐이다.

### 왜 윈도우에서 bypass 가 리눅스만큼 "안전"하지 않은가
리눅스 컨테이너는 외부격리(uid≠0, root 없음, 호스트 접근 차단)라 bypass 가 정당화됐다(codex 문서도
"externally sandboxed environments"로 이 플래그를 한정). **윈도우는 사용자 실기기다.** 그 외부격리
전제가 사라진다. 게다가 리눅스의 실제 안전벽(일회용 workdir·diff·rollout 감사)은 전부 bash
(`codex_run.sh`)에만 있고, 윈도우 주력 경로인 MCP 직접 통로는 그걸 안 거친다. 즉 **윈도우에서
그냥 bypass 를 켜면 = 사용자 홈 디렉터리 전체에 무승인·무샌드박스 자율 실행**이 된다. 절대 금지.

## 2. 현재 자산의 한계 (실측)

- **`codex_run.sh` 는 그린필드(빈 폴더) 전용이다.** `WD=mktemp .../runs/run_XXX` 로 **빈** 디렉터리를
  만들고 거기서 위임한다. 기존 코드베이스를 복사하지 않으므로 **"기존 파일 수정" 시나리오를 지원 못
  한다**(Issue 2). 새 파일 생성만 됨.
- **`PROJECT_ROOT` 가 리눅스 경로 하드코딩**(`/opt/data/projects/cc-plugin`). 윈도우에서 안 돎.
- **`codex_run.sh` 는 `codex exec`(헤드리스) 사용.** 윈도우 주력은 MCP(`codex mcp-server`) 통로라
  경로가 이원화돼 있다. 안전벽(diff·audit)이 exec 경로에만 있는 게 문제의 뿌리.

## 3. 설계 대안 3안 (남수장님 결정 필요: bypass 허용 범위)

공통 골격은 동일하다 — **격리 사본 + bypass 위임 + diff 리뷰 + 승인 후 apply**. 차이는
*bypass 를 어느 범위·어느 승인 강도로 허용하느냐*.

### 대안 A — 일회용 worktree 사본 + 자동 bypass + 사후 diff 승인 (리눅스 현행의 윈도우 이식)
- **흐름**: ①대상 repo 를 `git worktree add` 로 일회용 사본 생성(또는 임시폴더 복제) → ②그 사본 안에서만
  `sandbox=danger-full-access` 로 위임(기존 파일 수정 가능) → ③위임 후 `git diff` 캡처 →
  ④rollout 로그 audit(escape 탐지) → ⑤**사람이 diff 승인(P1)** → ⑥원본에 apply.
- **bypass 범위**: 일회용 사본 디렉터리 한정. 원본은 apply 전까지 안 건드림.
- **장점**: 리눅스와 통일된 멘탈모델. 기존 파일 수정 지원(worktree 는 실제 코드 포함). 자동화도 높음.
- **단점**: bypass 가 사본이라도 **사용자 홈 파일시스템 위에서** 돈다(윈도우엔 컨테이너 외벽이 없음).
  worktree 밖 절대경로 쓰기를 codex 가 시도하면 물리적으로 막을 게 없다(리눅스도 `-C` 는 경계가
  아니라고 실측됨 — audit 로그로 사후 탐지만 가능). → **사후 탐지 의존**이 윈도우선 더 위험.

### 대안 B — 일회용 worktree 사본 + on-request 승인 유지 + 대화형 게이트 (안전 최우선)
- **흐름**: 대안 A 와 동일하되 ②에서 `sandbox=workspace-write` + `approval_policy=on-request` 를
  쓴다. codex 가 쓰기/실행 직전 **사용자에게 승인을 묻는다**(대화형).
- **bypass 범위**: bypass 안 씀. 매 위험 동작마다 사람 승인.
- **장점**: 윈도우 실기에서 가장 안전. 무단 자율실행 원천 차단. bootstrap.ps1 이 이미 이 정책을 씀.
- **단점**: **실측상 workspace-write 는 윈도우서 read-only 강등돼 0파일일 수 있다**(§1). 즉 이 안은
  "안전하지만 아예 못 쓸" 위험. on-request 가 실제로 쓰기를 허용하는지 **윈도우 실기 검증이 선결**.
  (리눅스에선 bwrap 때문에 확인 불가 — 이게 우리가 못 닫는 지점.)

### 대안 C — 하이브리드: 사본에서 bypass 하되 "네트워크 차단 + 경로 화이트리스트"를 OS층에서 (정석)
- **흐름**: 대안 A + ②단계에 **OS 레벨 격리 추가** — 윈도우 방화벽 규칙으로 codex 프로세스 아웃바운드
  차단(모델 API 제외), 쓰기를 worktree 경로로 물리 제한(가능하면 별도 저권한 계정/컨테이너).
- **bypass 범위**: 사본 + OS 격리로 이중.
- **장점**: 근본 해결. 리눅스 컨테이너 외벽을 윈도우서 재현. 사후 탐지 의존 탈피.
- **단점**: 구현 비용 큼(윈도우 방화벽/ACL/저권한 계정 자동화). 사용자 환경마다 편차. 과할 수 있음.

## 4. 백팀장 추천

**대안 A 를 기본으로 하되, diff 승인(P1)을 반드시 사람 게이트로 강제 + rollout audit 를 윈도우에도
이식.** 이유:
- B 는 "안전하나 못 쓸" 실측 위험이 있어 윈도우 실기 검증 없이는 채택 불가.
- C 는 이상적이나 지금 급한 "윈도우에서 코딩 1건 돌리기"엔 과투자.
- A 는 리눅스에서 이미 검증된 3층 방어(격리 workdir + diff + audit)의 직수입이고, 윈도우 실기에서
  가장 빨리 "된다"를 만들 수 있다. **단 A 의 약점(사후 탐지 의존)을 남수장님이 수용하는지가 결정 포인트.**
- 장기적으로 C 의 네트워크 차단만이라도 얹으면 A 의 약점이 크게 준다 → **A로 시작, C의 방화벽 차단을 후속**.

## 5. 구현 시 열어야 할 세부(승인되면 그때 확정)
- worktree 생성/정리: `git worktree add`(원본 repo 필요) vs 임시폴더 전체복제(비-git 프로젝트 대응).
- 기존 파일 수정 시나리오: 사본에 실제 코드가 있어야 하므로 그린필드 `codex_run.sh` 를 "사본 기반"으로 개편.
- `PROJECT_ROOT`/경로: 리눅스 하드코딩 제거, 크로스플랫폼(`$PSScriptRoot`/`CLAUDE_PLUGIN_ROOT` 기반).
- 통로 일원화: 윈도우 MCP 경로에도 diff·audit 가 붙도록. bootstrap.ps1 이 심는 훅과 연계.
- P1(apply 전 diff 승인)·P2(위험패턴 강행 차단) 게이트를 양 OS 공통으로.
- doctor 3번(쓰기검증)이 이 경로 채택 후 PASS 로 바뀌는지 완료기준에 포함.

## 6. 리눅스에서 못 닫는 것 (윈도우 실기 필수)
- 대안 B 의 on-request 가 윈도우서 실제 쓰기를 허용하는지(리눅스 bwrap 때문에 확인 불가).
- 대안 C 의 윈도우 방화벽/ACL 자동화 실동작.
- worktree 사본에서 codex 가 사본 밖 절대경로 쓰기를 시도할 때 윈도우 물리 차단 가능 여부.
