01

경영 요약

Executive Summary

codex-harness는 소프트웨어 개발의 기획 → 결정 기록 → 코딩 → 검사 네 단계를 자동으로 진행하는 도구다. 오늘날 AI에 코딩을 맡길 때 반복되는 두 가지 손실 — 기획이 애매한 채로 개발이 시작돼 재작업이 발생하는 것, 그리고 AI가 "완료했다"고 보고하지만 실제로는 검증되지 않은 결과가 그대로 반영되는 것 — 을 구조적으로 차단한다.

도구는 먼저 요구사항을 끝까지 캐물어 확정하고, 그 결정을 기록해 이전 결정과 충돌하는지 점검한다. 이후 코딩은 별도의 AI에 맡기되, 결과물은 자동 검사와 사람의 승인을 통과해야만 실제 코드에 반영된다. "코드를 만드는 AI"와 "코드를 검사하는 AI"를 분리해 서로 교차 검증하는 것이 핵심 원리다.

권고: 도입 가능

사람이 최종 승인하는 반자동 방식은 지금 즉시 운영할 수 있다. 사람 없이 완전히 맡기는 무인 자동화는 격리 환경(별도 인프라)을 갖춘 뒤 권장한다.

02

배경과 목적 — 왜 필요한가

AI에게 개발을 맡기는 조직이 공통으로 겪는 두 가지 손실이 있다. 이 도구는 그 두 손실을 겨냥한다.

문제 1애매한 기획
  • 요구사항이 확정되지 않은 채 개발이 시작된다.
  • 중간에 "이게 아니었다"가 반복돼 재작업이 쌓인다.
  • 결정 근거가 남지 않아 나중에 왜 그렇게 했는지 모른다.
문제 2검증되지 않은 결과
  • AI가 "완료"라고 하지만 실제로 도는지는 별개다.
  • 겉보기엔 그럴듯하나 숨은 결함이 그대로 반영된다.
  • 사람이 일일이 확인하기엔 시간이 부족하다.
이 도구의 목적

기획은 확정될 때까지 넘어가지 못하게 막고, 결과물은 검사를 통과해야만 인정한다. 즉 두 손실이 발생할 여지 자체를 프로세스로 없앤다.

03

제품 개요 — 무엇인가

codex-harness는 클로드 코드(Claude Code)에 한 번 설치해두는 플러그인이다. 한 번 설치하면 이후 모든 개발 작업에 네 단계 절차가 자동으로 적용된다.

역할 분담두 AI의 협업
  • Claude — 기획을 확정하고, 결과물을 검사한다(관리자 역할).
  • Codex — 실제 코드를 작성한다(실무자 역할).
  • 만드는 AI와 검사하는 AI가 서로 다른 회사 제품이라 편들지 않고 교차 검증된다.
사람의 자리무엇만 하면 되나
  • "무엇을 만들지" 방향만 정한다.
  • 정말 갈리는 판단이 있을 때만 답한다.
  • 최종 결과를 코드에 반영할지 승인한다.
04

작동 방식 — 네 단계 절차

한 번의 지시가 아래 네 단계를 순서대로 통과한다. 앞 단계를 통과하지 못하면 다음 단계로 넘어가지 않는다.

1

기획 확정설계

도구가 면접관처럼 되묻는다. "무엇을, 왜, 어디까지, 어떤 방식으로." 다섯 가지 관점이 전부 명확해지기 전에는 "설계 완료"를 선언하지 못한다. 쉬운 것은 알아서 정하고, 정말 갈리는 것만 사람에게 묻는다.

2

결정 기록기록·점검

확정된 결정을 문서로 남기고, 이전에 내린 결정과 충돌하는지 점검한다. 충돌이 발견되면 코딩으로 넘어가지 못하게 막는다.

3

코딩 위임실행

확정된 기획을 바탕으로 Codex가 격리된 작업 공간에서 실제 코드를 작성한다. 결과는 "무엇이 어떻게 바뀌었는지" 변경 내역으로 정리돼 돌아온다.

4

검사와 승인검사

결과물은 세 가지 관문 — 문법 점검, 실제 동작 확인, 다른 AI의 교차 검토 — 을 모두 통과해야 한다. 마지막으로 사람이 승인해야 실제 코드에 반영된다.

핵심 원칙

코드를 만드는 쪽과 검사하는 쪽을 반드시 분리한다. 자기가 짠 코드를 자기가 검사하면 객관성이 없기 때문이다. 서로 다른 AI가 교차로 검증한다.

05

도입 방법 — 어떻게 쓰나

설치는 두 줄이면 끝난다. 이후는 대화하듯 "무엇을 만들지"만 말하면 된다.

# 1단계 · 설치 (최초 1회, 두 줄)
claude plugin marketplace add solitasroh/codex-harness
claude plugin install codex-harness@sol-dev

# 2단계 · 코딩 AI 연결 (최초 1회)
bash plugin/bin/codex_bootstrap.sh

# 3단계 · 이후로는 만들고 싶은 것을 말하면 된다
/codex-harness:harness-run "장바구니에 쿠폰 적용 기능"

이후 도구가 알아서 기획을 캐묻고, 결정을 기록하고, 코딩을 시키고, 검사한다. 사람은 갈리는 판단에 답하고 마지막에 승인만 하면 된다.

06

신뢰성 검증 — 믿을 수 있나

"만들었다"는 주장이 아니라, 실제로 확인한 결과만 싣는다. 아래는 모두 실측으로 검증된 항목이다.

실전 제작 검증

이 파이프라인으로 실제 도구 하나(.env 설정 파일 검증기)를 기획부터 검사까지 전 과정으로 만들어봤다. 결과물은 24개 검사 항목을 모두 통과했으며, 개발과 무관한 제3자가 독립적으로 다시 확인했다. 시연용 껍데기가 아니라 실제로 작동하는 도구를 만들어냈다.

자동 검사 통과 현황

검증 항목확인 방법결과
전체 자동 검사48개 항목 자동 실행48건 전원 통과
기획 확정 기능애매한 요청에 억지로 진행하지 않는지정상 차단 확인
결정 충돌 점검충돌·무충돌 양쪽 상황 실측양방향 정확
실전 제작물독립 제3자 재검증24건 전원 통과
검증 원칙

AI의 자기 보고는 그대로 믿지 않는다. 모든 결과는 사람이 직접 다시 돌려 확인한 것만 "검증됨"으로 인정했다. 이 보고서의 모든 수치는 실측 기반이다.

07

안전성과 한계 — 리스크는 무엇인가

코딩을 AI에 맡길 때 가장 큰 우려는 "위험한 명령을 실행하지 않을까"이다. 이에 대한 방어와 한계를 정직하게 밝힌다.

다층 방어 체계

위험한 작업을 막는 안전장치를 여러 겹으로 두었다. 가장 앞단에서 위험한 명령은 실행되기 전에 차단되며, 그 뒤로 실행 내역 감시, 결과물 검사, 사람 승인이 차례로 걸린다. 어느 한 겹이 놓쳐도 다음 겹이 잡도록 설계했다.

단계방어 내용작동 시점
1위험 명령 사전 차단 — 위험한 작업은 실행 직전에 막는다실행 전
2위험한 지시가 담긴 요청 자체를 거부위임 전
3실행 내역을 기록·감시해 이상 행동 탐지실행 중·후
4결과물 검사 + 사람의 최종 승인반영 전

잔여 리스크와 대응

리스크영향대응 방안
교묘히 숨긴 위험 행위텍스트 검사로 100% 탐지 불가사람의 결과 검토(반자동)로 보완
무인 완전 자동화사람 검토 없이는 사각지대 존재격리 전용 환경 구축 후 전환
외부 네트워크 접근격리 없으면 물리적 차단 불가네트워크 차단 컨테이너에서 실행

정직한 한계

반드시 알아둘 점

위 방어는 강력하지만 완벽한 원천 봉쇄는 아니다. 아주 교묘하게 숨긴 위험 행위까지 100% 잡지는 못한다. 따라서 사람이 결과를 검토하는 반자동 운영은 지금 안전하게 가능하지만, 사람 없이 완전히 맡기는 무인 자동화를 하려면 별도의 격리 환경(외부와 차단된 전용 공간)을 갖추는 것이 전제 조건이다. 이 부분은 과장 없이 있는 그대로 밝힌다.

08

결론과 권고 — 지금 무엇을 할 수 있나

도입 판단에 필요한 결론을 두 갈래로 나눠 제시한다.

지금 가능

사람 승인형 반자동 운영

사람이 최종 결과를 검토·승인하는 방식으로는 지금 즉시 도입해 쓸 수 있다. 기획 누락과 검증 누락을 막는 효과를 바로 얻는다.

조건부

완전 무인 자동화

사람 개입 없이 맡기려면 격리 전용 환경(외부와 차단된 인프라)을 먼저 갖춰야 한다. 이는 서버 관리 차원의 작업으로, 도입 여부는 별도 판단이 필요하다.

한 줄 결론

codex-harness는 완성되어 지금 쓸 수 있는 도구다. 사람이 방향과 승인을 맡는 조건에서 기획·검증의 품질을 구조적으로 끌어올린다.
