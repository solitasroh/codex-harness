§00 — OverviewPM · 자비서

한 장 요약

비개발자도 이해할 수 있는 한 문단.

이 도구는 클로드 코드에 한 번만 설치해두면, 새 프로그램을 만들 때마다 설계 → 기록 → 코딩 → 검사 네 단계를 자동으로 밟아준다. 먼저 클로드가 면접관처럼 질문을 던져 "무엇을 만들지"를 빈틈없이 확정하고, 그 결정을 문서로 남긴 뒤, 실제 코드 작성은 코덱스—다른 AI—에게 맡긴다. 마지막으로 그 코드가 제대로 됐는지 세 관문으로 검사해, 통과한 것만 "완료"로 인정한다.

A무엇을

기획·기록·코딩·검사를 한 번에 굴리는 클로드 코드 플러그인.

B누가 쓰나

아이디어는 있는데 구현 디테일과 검증이 늘 빠지는 팀 또는 1인 개발자.

C왜 안전한가

AI가 "다 됐다"는 말을 그대로 믿지 않고 증거로 검사. 위험엔 반드시 사람이 확인.

§01 — User PerspectiveDesign · 엘레나

사용자 관점 — 이걸로 무엇을 하나

설치부터 실제 사용 흐름까지, 개발을 몰라도 따라올 수 있게.

3분 만에 시작

# 플러그인 로드 (클로드 코드)
claude --plugin-dir ./plugin

# 코덱스 인증 (최초 1회)
bash plugin/bin/codex_bootstrap.sh

# 코딩 위임 — 설계·ADR 확정 후
bash plugin/bin/codex_run.sh "만들고 싶은 것"

실제 사용 흐름 — "알림 기능 만들어줘"라고 하면

1

클로드가 되묻는다

"어떤 채널로 보내나요? 인앱·푸시·이메일? 읽음 처리는요?" 애매한 게 남으면 다음으로 넘어가지 않는다. 쉬운 건 알아서 정하고, 정말 갈리는 것만 묻는다.

2

결정을 남긴다

정해진 것을 문서로 기록하고, 이전 결정과 부딪히지 않는지 검사한다. 충돌하면 코딩으로 넘어가지 못한다.

3

코덱스가 코딩한다

격리된 작업 공간에서 코덱스가 실제 코드를 쓴다. 결과는 "무엇이 바뀌었는지" 변경 내역으로 정리돼 나온다.

4

검사 후 완료

세 관문(문법·동작·교차검토)을 증거로 통과해야 "완료". 마지막 반영은 사람이 승인할 때만.

Key takeaway

"무엇을 만들지"만 정하면 된다. 캐묻고 기록하고 코딩하고 검사하는 지루한 과정은 도구가 대신하고, AI가 몰래 대충 하고 "다 됐다"고 우기는 것을 막아준다.

§02 — How It WorksDesign · 엘레나

네 단계 파이프라인

한 번의 지시로 아래 네 단계가 순서대로 흐른다.

1

설계design-grill

면접관처럼 캐물어 요구사항 확정. 다섯 관점(목적·완료기준·범위·대안·가정) 미해소 0개 전엔 완료 불가.

2

ADR 확인adr-check

결정을 기록하고 기존 결정·용어집과 충돌 대조. 부딪히면 코딩 진입 차단.

3

코덱스 코딩codex mcp-server

클로드가 지휘하고 코덱스가 격리 공간에서 코딩. 변경 내역을 회수한다.

4

검사G1 · G2 · G3

문법·경계 → 실제 동작 → 교차검토. 세 관문을 증거로 통과해야 완료.

크로스벤더 원칙

코딩(Codex)과 검토(Claude)는 반드시 다른 AI가 맡는다. 자기가 짠 코드를 자기가 검토하는 self-review는 무효 — 서로 다른 벤더가 교차 검증한다.

정직한 한계 — 숨기지 않는다

훅과 로그 감사는 "명령 텍스트" 기반이라, 인터프리터 내부 유출 (예: python3 -c 'open(...).write(...)') 같은 교묘한 변형은 잡지 못한다. 즉 지금 방어는 강력한 탐지·사전차단이지 완전 차단막은 아니다. 완전 무인 자동화의 유일한 근본 경계는 컨테이너 격리(호스트 작업)이며, 그 전까지는 사람 검토 전제 반자동으로 안전하게 운영한다.

§04 — SafetyDevLead · 백팀장

안전 경계 — 다섯 겹의 방어

codex는 claude와 별도 프로세스라 claude 안전벨트가 자동 적용되지 않는다. 그래서 다층으로 막는다 — 어느 하나도 단독 완결이 아니다.

1선

codex PreToolUse 가드 훅 — claude 가드 이식

codex가 위험 명령을 실행하기 직전 스스로 차단. 사후 추적이 아니라 실행 전 차단 — 이 프로젝트의 전환점.

실행 전 차단
2선

위임 프롬프트 사전 스캔

위험 지시가 담긴 코딩 요청 자체를 위임 전에 거부한다.

위임 전
3선

실행 중 rollout 로그 감사

codex가 실제 실행한 모든 명령을 로그에서 검사. 변경 내역에 안 남는 행위까지 포착·표면화.

실행 후 탐지
4선

변경 내역 + 검사 게이트 + 사람 승인

결과물을 세 관문으로 검사하고, 코드베이스 반영은 사람이 승인할 때만.

결과물 게이트
근본

컨테이너 네트워크·파일시스템 격리

완전 무인 자동화 시 유일한 물리적 차단. 호스트 작업으로, 완전 자동화로 갈 때 도입.

호스트 · 선택
§05 — VerificationQA · 브라이언

검증·신뢰성 관점 — 어떻게 믿나

"AI 자기보고 불신 → 실제 증거로 판정" 원칙이 어떻게 관철됐나.

?측정기부터 의심

"통과했나"보다 "측정 도구가 진짜 대상을 재는가"를 먼저 본다. 방어를 일부러 부숴서 테스트가 실패(RED)하는지 확인 — 빈 껍데기가 아님을 증명한다.

✕결함 소거 여정

로그 감사기를 계속 뚫어보며 미탐·오탐·통과누수 결함(FN1–FN5-D)을 순차 발견·수정. "오탐 고치다 미탐 만들지 않기" 원칙 관철.
