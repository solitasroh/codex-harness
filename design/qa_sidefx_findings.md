# QA 소견: 사후검사(diff) 모델의 구멍 + 실행 로그 감사 보완

작성: 브라이언(QA) | 2026-07-05 | 회의: codex 격리·사후검사 모델 점검 (수장 질문)
안건: ① 실행 중 부작용을 사후 diff로 못 잡는 문제 ② qa_verify가 "실제 한 일" vs "diff에 남은 것" 중 무엇을 측정하나

---

## 결론: PM 진단이 맞다. diff 검사는 "행위"가 아니라 "결과 상태"만 본다. **단, 구멍을 메울 근거(codex 실행 로그)가 이미 존재한다.**

---

## ① qa_verify가 실제로 측정하는 것 (코드로 확정)

- `qa_verify.sh` L17: `git diff --cached --name-only` — **workdir 안에서 git이 추적한 파일의 최종 변경**만 봄.
- `codex_run.sh` L36~40: 역시 `git -C $WD diff --cached` — 동일.
- → **측정 대상 = "diff에 남은 것"이지 "codex가 실행 중 실제 한 일"이 아니다.** PM 우려 정확.

## ② diff가 못 보는 부작용 — QA 무해 프록시 실증

임시 workdir에서 codex가 실행 중 했을 법한 부작용 3종을 무해하게 모사 → diff 관찰:

| 부작용 | diff에 남나 | 이유 |
|--------|:----------:|------|
| 임시파일 생성 후 삭제(create-then-delete, 예: 시크릿 유출용 임시파일) | ❌ 안 남음(0건) | diff는 baseline↔최종 상태 차분. 중간에 생겼다 사라진 건 안 보임 |
| workdir **밖** 읽기(`/etc/hostname` 등 외부 정보 접근) | ❌ 남을 수 없음 | 읽기는 파일 변경이 아님 |
| 외부 네트워크 호출(데이터 유출/악성 다운로드) | ❌ 남을 수 없음 | 네트워크는 파일시스템 diff에 안 나타남 |

**실측 결과: 3종 전부 diff에 흔적 0.** result.txt(정상 산출물)만 diff에 남음. → "사후 diff만으론 실행 중 파괴/유출을 못 잡는다"가 실증됨.

## ③ 구멍을 메울 근거 — codex 실행 로그 (핵심 발견)

codex는 실행마다 `.codex_home/sessions/**/rollout-*.jsonl`에 **`exec_command` 이벤트로 실행한 모든 셸 명령을 기록**한다(실측 17건 확인, 과거 canary 명령까지 전부 남아있음).
- **diff엔 없는 "행위"가 로그엔 있다.** 즉 create-then-delete, 외부 읽기 명령, `curl`/`wget` 네트워크 호출이 여기 남는다.
- → **사후 감사 대안이 이미 데이터로 존재.** 우리가 안 쓰고 있었을 뿐.

## QA 권고 — 다층 방어 (사후검사만으론 부족, 3겹으로)

### (a) 사전(prevention) — 가능한 곳은 실행 전 차단
- codex 위임 **프롬프트를 실행 전 스캔**: 프롬프트 자체에 위험 지시(외부 유출, workdir 밖 경로) 있으면 거부. (백팀장 ② 항목과 연결)
- 근본 해법은 OS 샌드박스지만 이 컨테이너는 bwrap 불가(기확인). → 아래 (b)(c)로 보완.

### (b) 실행 중 감사(detection) — rollout 로그를 검사에 편입 ★신규 제안
- qa_verify에 **L1.5 로그 스캔 단계 추가**: 해당 run의 `rollout-*.jsonl`에서 `exec_command`를 뽑아 위험 패턴(danger_patterns.txt) 대조.
- 잡을 것: workdir 밖 절대경로 쓰기/삭제, `curl|wget ... | sh`, 외부 IP 접속, create-then-delete.
- **이게 diff 사각지대를 메우는 핵심.** diff는 "무엇이 남았나", 로그는 "무엇을 했나" — 둘 다 봐야 완전.

### (c) 격리 강화 — 부작용 범위 물리적 제한
- codex를 **네트워크 차단된 환경**에서 실행(가능하면 netns/방화벽). 최소한 실행 중 아웃바운드를 로그로라도 감지.
- workdir 밖 쓰기: `-C $WD`는 cwd만 고정, 절대경로는 못 막음(백팀장 ① 실측 대기). → 로그 스캔(b)으로 사후 포착.

## QA 판정
- **현 모델(격리 workdir + diff + 사람승인)은 "결과물 오염"은 잘 막지만 "실행 중 행위"엔 사각지대.** 수장 질문이 정확히 이 지점을 짚음.
- **최우선 보완 = (b) rollout 로그를 qa_verify에 편입.** 이미 있는 데이터라 구현 비용 낮고 사각지대를 직접 메움. Phase 2 필수 항목으로 등록 권고.
- 이건 내 골든셋 메타검증 교훈의 연장: **"측정기가 대상을 측정하나."** qa_verify는 지금 "결과"만 측정 중 → "행위"까지 측정하도록 확장해야 완전한 게이트.
