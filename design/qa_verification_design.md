## 0. 설계 원칙 (한 줄)

> **"코딩 완료"는 AI가 선언하는 게 아니라, 게이트가 증거로 판정한다. 증거 없으면 미완료.**
> 통과만 확인하는 건 반쪽 — 나쁜 코드를 넣었을 때 반드시 막히는 것까지 확인해야 게이트다(fail-closed).

---

## 1. "코딩 완료" 판정 — 3단계 게이트 합격 기준 (L3 v1 발전)

코딩 결과물은 아래 G1→G2→G3 순서로 통과해야 "완료". **하나라도 미통과 = 재작업.** 순서 중요(싼 검사 먼저, 비싼 교차검토 마지막).

### G1. 문법·경계 (기계 자동, 즉시)
| 항목 | 합격 기준 | 불합격 시 |
|------|----------|----------|
| 문법 | 모든 소스 파일이 컴파일/파싱 성공 | 즉시 재작업(Critical) |
| 변경 경계 | 변경 파일이 전부 격리 작업공간(workdir) 안 | 밖 변경 1건이라도 = 즉시 중단(Critical) |
| 산출 존재 | 위임 프롬프트가 요구한 파일이 실제 생성됨 | 미생성 = 재작업 |

### G2. 실제 동작 (테스트를 진짜 돌림)
| 항목 | 합격 기준 | 비고 |
|------|----------|------|
| 테스트 실행 | 테스트를 **실제 실행**(파일 존재만으론 불충분) | pytest 있으면 pytest, 없으면 함수 직접호출 폴백 |
| 회귀 0 | 변경 전 기준(baseline) 대비 **신규 실패 0** | 기존 실패는 카운트 제외, 신규만 차단 |
| 요구사항 매핑 | 스펙의 각 요구항목이 테스트/동작으로 **항목별 확인** | 매핑 안 된 요구 = 미완료 |
| 경계·예외 | happy path 외 **빈입력·잘못된입력·경계값** 최소 1케이스 | QA가 케이스 정의 |

### G3. 교차검토 (다른 AI + 사람 게이트)
| 항목 | 합격 기준 | 근거 |
|------|----------|------|
| 다른 벤더 검토 | 코딩=Codex면 **검토=Claude**(다른 AI). 자기검토 무효 | 영상2 크리틱 원칙 |
| 결함 주입 생존성 | 핵심 로직에 자동 변이(연산자 뒤집기 등) 주입 시 테스트가 **RED**. GREEN이면 테스트가 결함을 못 잡음 = 불합격 | phase1 fail-closed 방식 경량화 |
| 보안 스캔 | 위험 패턴(아래 2절) 히트 시 **기본 fail-closed** | — |
| 의도 일치 | diff가 위임 프롬프트 범위와 일치. 범위 초과 변경 = 반려 | — |
| 판정 파싱 | 검토 AI 응답이 파싱 불가/모호 = **불합격 처리**(안전측) | fail-closed |

**최종 판정 규칙**: `완료 = G1 ∧ G2 ∧ G3`. 판정 근거로 **실제 명령 출력·종료코드·diff·검토 결과**를 첨부. 자기보고 문장만 있으면 자동 미완료.

## 2. 위험 코드 차단 규칙 (실측 수정본 반영)

diff의 **추가된(+) 라인만** 검사(삭제는 위험 유입 아님). 기본은 정보성(리포트, exit 0), `--strict`에서 히트 시 차단(exit 2).

### 현재 위험 패턴 (run_Oz3JuL 확정본)
| 분류 | 패턴 | 잡는 것 |
|------|------|---------|
| 임의 실행 | `eval(`, `exec(`, `os.system` | 문자열을 코드로 실행 |
| 셸 호출 | `subprocess\.(run\|call\|Popen\|check_output\|check_call)` | **실행 호출부만**(정상 import 오탐 제외 — D-Q1 수정) |
| 파괴 명령 | `rm -rf` | 강제 삭제 |
| 위험 플래그 | `--dangerously`, `--yolo` | 승인/샌드박스 우회 |
| 시크릿 | `(API_KEY\|SECRET\|PASSWORD\|TOKEN)\s*=` | 자격증명 하드코딩 |
| 원격 실행 | `(curl\|wget)\b.*\|\s*(?:sudo\s+)?(?:ba)?sh` | 인터넷 스크립트를 셸로 파이프(**wget·공백·bash·sudo 변형 포괄** — D-Q2 수정) |

### QA가 요구하는 운영 규칙
1. **회귀 고정 케이스 필수**: 방금 수정한 D-Q1(`import subprocess`는 안 잡음)·D-Q2(`wget|sh`·`curl |  sh`는 잡음)를 **테스트에 고정 등록**. 안 하면 패턴을 누가 건드릴 때 조용히 재발. → 플러그인 제작 시 첫 작업.
2. **미탐 > 오탐 우선순위**: 오탐은 사람이 거르지만 미탐은 위험이 통과함. 패턴 추가 시 미탐부터 막는다.
3. **패턴은 데이터로 분리**: 코드 하드코딩 말고 목록 파일로 → 프로젝트별 확장/감사 가능.
4. **차단은 opt-in**: 기본 정보성(리포트만), 게이트로 쓸 땐 `--strict` 명시. 오탐이 파이프라인을 무단 차단하지 않도록.

## 3. 사람 승인이 반드시 필요한 지점 (설계 명문화)

자동화하되, 아래는 **사람(수장/리드)의 명시 승인 없이 통과 금지**. 무인 자율은 여기서 멈춘다.

| # | 승인 지점 | 왜 사람이 필요한가 | 기본 동작 |
|---|-----------|-------------------|-----------|
| P1 | **코드베이스로 apply 전** | 격리 workdir 결과를 실제 프로젝트에 반영하는 순간이 진짜 위험. diff를 사람이 보고 승인 | diff 제시 → 승인 대기(자동 apply 금지) |
| P2 | **위험 패턴 히트를 무시하고 진행** | 스캐너가 위험을 잡았는데 강행하려면 사람이 근거 대고 판단 | `--strict` 차단 → 사람이 명시 해제해야 진행 |
| P3 | **위험 플래그 사용**(`--dangerously-bypass` 등 샌드박스/승인 우회) | 이 서버는 내장 샌드박스를 못 켜므로 이 플래그가 불가피하나, 그 자체가 고위험 결정 | 설계상 수장 1회 승인된 경계 안에서만. 신규 확대는 재승인 |
| P4 | **설계 단계 진짜 갈리는 결정**(grill me의 [NEEDS 사용자]) | 요구사항이 근본적으로 갈리는 지점은 AI가 추측하면 안 됨 | 자동 해소 불가분만 사용자에게 질문(쉬운 80%는 자동) |
| P5 | **게이트 2회 재작업 후에도 미통과** | 무한 루프 방지. 사람이 방향 재설정 | 2회 실패 → 중단·사람 에스컬레이션 |

원칙: **P1(apply)과 P2(위험 무시)는 절대 자동화 금지.** 나머지는 정책으로 조정 가능하되 기본은 안전측.

## 4. 이 게이트를 코드로 (백팀장 연결점)
- 이미 만든 `qa_verify.sh` = G1·G2 자동화 실물. 플러그인 `bin/`에 편입 권고.
- `codex_run.sh` 끝의 diff 캡처 = P1 승인 자료. `hooks/`의 PreToolUse로 P2·P3 차단 훅 검토.
- G3(교차검토)는 오케스트레이터(Claude) 스텝 — 아직 수동, Phase 2에서 자동화. 합격 기준은 위 G3 표.
- 결함 주입 생존성(G3)은 `qa_failclosed_check.py` 방식을 경량 자동 변이로 발전.
