# QA 검증 기준 — design-grill / adr-check 스킬

작성: 브라이언(QA) | 2026-07-05 | 대상: `plugin/skills/design-grill/SKILL.md`, `plugin/skills/adr-check/SKILL.md`
용도: 엘레나·백팀장 산출물이 오면 이 기준으로 검증. 산출 전 기준 선정의(측정 편향 방지).

---

## 0. QA 대전제 — 스킬은 코드가 아니라 프롬프트다

> 지금까지 검증한 것(스캐너·훅)은 **코드 = 결정적**이라 "결함 주입 → RED"가 명확했다.
> 스킬은 **프롬프트 = 확률적**이다. 같은 지침도 모델·문맥·온도에 따라 다르게 행동한다.
> **"1회 통과 = 항상 통과"가 성립하지 않는다.** 이 한계를 숨기고 "게이트 실효 확인 ✅"라 보고하면
> 그거야말로 QA가 가장 경계하는 "확인 안 된 걸 된 것처럼" 하는 것이다.

그래서 검증을 **3층**으로 나눈다. 정적층은 결정적으로 못 박고, 행위층은 골든 시나리오로 표본 확인하며, 비결정성은 정직하게 경계로 남긴다.

---

## L1. 정적 검증 (결정적, grep 가능) — "빈 껍데기가 아닌가"

PM이 걱정한 "뼈대만 있고 알맹이 없는" 스킬을 잡는 층. 자동 스크립트로 판정.

### 공통 (두 스킬 모두)
| # | 기준 | 판정 |
|---|------|------|
| S1 | 유효한 SKILL.md 구조(frontmatter `name`+`description`) | 없으면 FAIL |
| S2 | 트리거 조건 명시(언제 이 스킬이 켜지는가) | 없으면 FAIL |
| S3 | 절차가 번호/단계로 구체화(추상 서술 아님) | 4단계 미만 = 경고 |
| S4 | 산출 포맷 정의(다음 단계로 넘길 결과물 형태) | 없으면 FAIL |
| S5 | **예시 1개 이상**(프롬프트는 few-shot 예시가 실효성 좌우) | 없으면 경고 |

### 게이트 강제성 (핵심 — 정적으로 측정하는 대리 지표)
게이트가 "실효"하려면 우회 여지를 안 줘야 한다. 텍스트의 **강제 표현 vs 물렁 표현** 비율로 측정:
- **강제 표현(있어야 함)**: `반드시`, `까지`, `금지`, `않으면 진행`, `없이 …선언 금지`, `모든 …해소`, `MUST`, `절대`
- **물렁 표현(게이트를 무르게 함, 감점)**: `가능하면`, `권장`, `되도록`, `보통`, `선택적으로` — 게이트 지점에 이런 말이 붙으면 우회 가능.

| # | 기준 | 판정 |
|---|------|------|
| G-static | 게이트 문장에 강제 표현 ≥1, 게이트 지점에 물렁 표현 0 | 위반 시 "게이트 무름" 경고 |

## L2. 행위 검증 (골든 시나리오, 표본) — "게이트가 실제로 작동하는가"

정적으로 "강제 문구가 있다"는 확인해도, **실제로 그렇게 행동하는지**는 실행해야 안다. 골든 시나리오 세트로 표본 확인.

### 설계 스킬(design-grill) — 게이트 실효 골든셋
| ID | 입력 시나리오 | 기대 행동 | 이러면 FAIL |
|----|--------------|-----------|-------------|
| DG-1 | 모호한 요구("로그인 기능 만들어줘", 인증방식·저장소 미정) | **질문한다**(다음 단계로 안 넘어감) | 스펙을 지레짐작하고 진행 |
| DG-2 | 5차원 중 1개만 미해소 상태에서 "그냥 진행해" | 미해소 차원을 짚고 확인 요청 | 모호한 채 확정 스펙 산출 |
| DG-3 | 코드베이스로 답 가능한 질문(기존 DB 종류 등) | 사용자에 안 묻고 코드 탐색 | 불필요하게 사용자에 질문(DX 위반) |
| DG-4 | 전 차원 해소 완료 | 확정 스펙 산출 + 다음 단계 진행 | 계속 질문(무한 루프) |

핵심 게이트: **DG-1, DG-2** = "모호하면 안 넘어간다". 이게 뚫리면 게이트 실패.

### ADR 스킬(adr-check) — 충돌 감지 골든셋
| ID | 기존 ADR | 새 결정 | 기대 | 이러면 FAIL |
|----|---------|--------|------|-------------|
| ADR-1 | "언어=Python stdlib only"(ADR-001) | "rust로 재작성" | **충돌 감지 + 사용자 알림** | 조용히 통과(미탐) |
| ADR-2 | "위험히트 non-blocking, --strict opt-in"(ADR-002) | "위험히트면 무조건 차단" | 충돌 감지 | 미탐 |
| ADR-3 | (무관한 기존 ADR들) | 완전히 새 영역 결정 | 충돌 아님(통과) | 오탐(무관한데 충돌이라 함) |
| ADR-4 | 기존 결정 재확인 | 같은 결정 반복 | 충돌 아님(멱등) | 오탐 |

핵심: **ADR-1, ADR-2 미탐 = 치명(Major).** ADR-3, ADR-4 오탐 = Minor(과잉 경보). **미탐 > 오탐 우선순위**(스캐너와 동일 원칙).

> **검증 방식**: 백팀장 충돌감지가 **코드(스크립트)면** → 골든셋으로 결정적 fail-closed 테스트(충돌 심어 잡는지, 방어 빼면 RED). **순수 프롬프트면** → 골든 시나리오를 실제 로드해 표본 실행, 통과율 기록(단, L3 한계 명시).

---

## L3. 비결정성 경계 (정직한 한계 — 숨기지 않는다)

- 프롬프트 스킬의 골든 시나리오 통과는 **표본**이지 증명이 아니다. "5/5 통과"라도 다른 표현의 모호 입력엔 뚫릴 수 있다.
- 그래서 **정적 게이트 강제성(L1)을 실효성의 1차 대리 지표**로 삼는다. 강한 강제 문구 = 뚫릴 확률↓.
- 행위(L2)는 "적어도 이 대표 케이스에선 작동함"을 보이는 것. **회귀 골든셋으로 고정**해 스킬 수정 시 재검(코드 회귀와 동일 정신).
- 보고 시 반드시 명시: "정적 게이트 확인 + 골든 N/N 표본 통과. 단 프롬프트 특성상 100% 보장 아님."

---

## 판정 규칙
- **PASS**: L1 FAIL 0 + L2 핵심 게이트(DG-1/2, ADR-1/2) 전부 통과.
- **조건부**: L1 경고만 있거나 L2 비핵심(DG-3, ADR-3/4) 일부 미달 → 보강 후 진행.
- **FAIL**: L1 FAIL 있거나 L2 핵심 게이트 뚫림 → 재작업.
