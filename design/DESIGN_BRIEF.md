# 설계서 회의 브리프 — "설계→ADR→코덱스 코딩" 클로드 코드 플러그인

작성: 자비서(PM) | 2026-07-05 | 목적: HTML 설계서를 만들기 위한 회의 안건 + 뼈대
수장 확정 목표: **클로드 코드에 한 번 설치하면 [grill me 설계 → ADR 확인 → 코덱스 코딩 → 검사]를 자동으로 굴리는 플러그인.**
(diff-digest는 파이프라인 작동 확인용 연습작이었고, 진짜 목표는 이 플러그인 자체.)

---

## A. 이미 확정된 사실 (회의에서 재론 불필요 — 근거로만 사용)

- **코딩 실행 방식 = MCP(codex mcp-server) 주력 + codex exec 보조.** 플러그인(A)은 검증 참조용만. (실측 완료)
- **안전 경계 = 4겹**(컨테이너 외부격리 + 일회용 workdir + git diff 게이트 + 검사 자동화). 이 서버는 codex 내장 샌드박스를 물리적으로 못 켜므로 이 경계로 대체. (수장 승인 대기였으나, 설계서엔 "이 방식으로 설계"로 반영)
- **검사 방식** = 코드가 (1) 문법·경계 통과 (2) 테스트 실제 통과 (3) 다른 AI가 교차검토, 3단계 다 통과해야 완료. 나쁜 코드는 통과 못 시킴(실증됨).
- **참조 아키텍처** = max4c/skills(grill-me, write-prd, grill-with-docs, tech-spec, cookoff, verify-before-done). RobMitt/grill-me-skill 실물 확인.

## B. 설계서에 담아야 할 것 (HTML 문서 목차안)

1. **한 장 요약** — 이 플러그인이 뭐고, 누가 왜 쓰는가 (쉬운 말)
2. **전체 흐름도** — 설계(grill me) → ADR 확인 → 코덱스 코딩 → 검사, 4단계 파이프라인 그림
3. **각 단계 상세**
   - 3-1 설계(grill me): 클로드가 질문으로 요구사항 확정. 쉬운 80% 자동, 진짜 갈리는 것만 사용자에게(엘레나 DX 원칙)
   - 3-2 ADR 확인: 결정 기록·용어집 대조·인라인 갱신
   - 3-3 코덱스 코딩: MCP로 코덱스 호출, 격리 작업공간, 결과 diff
   - 3-4 검사: 3단계 게이트, 통과해야 완료
4. **플러그인 구조** — .claude-plugin/plugin.json + skills/ + hooks/ + bin/ (어떤 파일이 들어가는가)
5. **안전장치** — 4겹 경계, 위험 코드 차단, 사람 승인 지점
6. **설치·사용법** — 한 번 설치 후 어떻게 쓰나
7. **만드는 순서(단계)** — 무엇부터 만들지
8. **역할 분담** — 누가 뭘 맡나

## C. 회의 파트 배분 (각 팀장 담당)

- **백팀장(개발/기술설계)**: 위 3-3, 4, 5의 기술 부분 — 플러그인이 실제로 어떻게 구성되고(파일 구조), 코덱스를 MCP로 어떻게 호출하며, 4단계를 코드로 어떻게 엮는지 설계.
- **브라이언(QA/검증설계)**: 위 3-4 상세 — "코딩 완료" 판정 3단계 게이트의 합격 기준, 위험 코드 차단 규칙, 사람 승인 지점을 검증 관점에서 설계.
