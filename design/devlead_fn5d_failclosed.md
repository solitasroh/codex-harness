# DevLead — FN5-D 수정: cwd 필터 fail-open → fail-closed (브라이언 보안 지적)

작성: 백팀장(DevLead) | 2026-07-06 | 대상: 브라이언 QA FN5-D(cwd 필터가 fail-open 유발)

## 브라이언 지적 = 정확한 보안 결함 (수용)
FN5-C(이웃 run 오탐)를 잡으려 넣은 cwd 필터가 **로그를 스킵**해서, cwd 불일치 로그에 escape가
있어도 "감사할 로그 없음 → exit 0 통과"라는 **더 위험한 미탐(fail-open)**을 만들었다.
- 실측 재현: escape 명령 + session_meta.cwd 불일치 로그 → 필터가 로그 스킵 → exit 0. 유출 새는데 GREEN.
- 이는 내가 계속 지켜야 할 **미탐 > 오탐** 원칙 위반. 오탐 수정이 미탐으로 넘어감.

## 수정 — "스킵" 폐지, "분류"로 (fail-closed)
cwd 필터로 로그를 **버리지 않는다.** 시간창 내 로그를 (matched / unmatched)로 분류:
- **matched** (session_meta.cwd == 이 run workdir) = 이 run 확정 → 위험이면 **하드차단(exit 2)**.
- **unmatched** (cwd 불일치/특정불가) = 이 run인지 불확실 → **위험 발견 시 opaque(exit 3)로 표면화**.
  스킵 안 함. "이 run 로그를 특정 못 함"도 사각지대이므로 버리지 않고 사람에게 올린다.
- **로그 0건 + --delegated** = 위임했는데 로그 없음 = 이상신호 → opaque(exit 3). (FN5-D #2)
- cwd 매칭에 **realpath 정규화**(심링크·슬래시 흡수, FN5-D #3).

## 설계 판단: unmatched+정상은 무시 (미탐>오탐 + 실용성 양립)
- unmatched 로그가 **위험**이면 표면화(3), **정상**이면 무시(0).
- 근거: fail-open 우려의 핵심은 "위험이 스킵되는 것" — 그건 위험 unmatched 표면화로 이미 막음.
  정상 unmatched까지 매번 플래그하면 이웃 정상 run마다 opaque가 떠서 FN5-C 오탐이 부활(실용성 death).
- 즉 **불일치+위험=표면화, 불일치+정상=무시**. 미탐 방지(위험은 절대 안 버림) + 오탐 억제 양립.

## 실측 (전부 회귀 고정)
| 케이스 | 기대 | 결과 |
|--------|------|------|
| matched cwd + 정상 | 통과(0) | ✅ (FN5-C 오탐 없음) |
| matched cwd + escape | 하드차단(2) | ✅ |
| **unmatched cwd + escape** | **표면화(3)** | ✅ **FN5-D fail-open 해결** |
| 로그 0건 + delegated | 이상신호(3) | ✅ |
| unmatched cwd + 정상 | 무시(0) | ✅ (실용성) |
- 전체 코드 회귀: **38/38**(scan 10 + hook 8 + audit 20). 폴백 20/20. 문법 OK.

## 교훈 (브라이언 원칙 계승)
- **오탐 수정이 미탐 방향으로 넘어가면 안 된다.** 측정기가 대상을 "못 볼 때"의 기본값이 GREEN이면
  그게 제일 위험 — 애매하면 통과가 아니라 사람에게 표면화(fail-closed).
- cwd 필터의 올바른 역할 = "이 run 확정(하드차단 대상) 선별"이지 "나머지 폐기"가 아님.
  확정 못 한 것은 버리지 말고 낮은 심각도(리뷰 플래그)로 올린다.
- 근본 결론 불변: 로그 감사=탐지 보조, 진짜 경계=P1 사람검토 + **컨테이너 격리**(수장 최우선 결정).
  격리가 있으면 감사기가 로그를 놓쳐도 유출이 물리 차단 — 이번 건이 격리 필요성을 또 방증.
