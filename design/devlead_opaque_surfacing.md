# DevLead — FN4 대응: opaque 표면화 (브라이언 방향 동의 + 최소 조치)

작성: 백팀장(DevLead) | 2026-07-06 | 대상: 브라이언 QA 재검증 FN4 + "격리가 근본" 소견

## 브라이언 방향에 전적 동의
- FN4(인터프리터 one-liner 유출) 재현 확인: `python3 -c 'open(...).write(open("/etc/passwd").read())'`,
  `perl -e`, `base64|sh` 전부 미탐이었음. 실측 재현 완료.
- 브라이언 판단 옳음: **"명령 텍스트 정규식으로 유출 탐지"는 근본 상한**. 유출 변형은 무한(두더지잡기).
  개별 유출 정규식을 계속 추가하는 방향은 QA와 함께 **폐기**. 근본 = 컨테이너 격리 + 사람 검토(P1).

## 그래서 한 것: 정규식 확장이 아니라 opaque 표면화 (다른 층위)
개별 유출을 쫓는 대신, **"로그 텍스트만으로 행위를 알 수 없는 실행"을 만나면 자동 GREEN 금지**라는
단일 fail-closed 메타규칙을 넣음. 이건 두더지잡기가 아니라 브라이언 소견 #2("로그 감사 통과=안전
과신 금지")를 코드로 박는 것.
- 감지: 인터프리터 `-c/-e`(python/perl/ruby/node/php/lua) + I/O 신호(open/write/socket/urllib/
  subprocess/base64 등), 또는 `base64|xxd -d | sh/python` 난독 실행.
- 처리: **차단(exit 2) 아님, 리뷰 플래그(exit 3).** codex_run은 non-zero면 apply 막으므로 결과적으로
  "자동 통과 금지 + 사람 검토(P1)". 즉 "감사가 놓칠 수 있는 실행"을 사람 앞에 표면화.

## 실측 (test_audit_escape.py, 15/15)
- opaque 3종(python -c I/O, perl -e, base64|sh) → **exit 3 표면화** ✅
- 명시 escape(cp /etc/passwd) → 여전히 exit 2 차단 ✅
- 인터프리터라도 I/O 없으면(`python3 -c "print(2+2)"`) opaque 아님 → exit 0 (오탐 방지) ✅
- 전체 코드 회귀: **33/33**(scan 10 + hook 8 + audit 15).

## 정직한 트레이드오프 (숨기지 않음)
- opaque는 **의도적으로 보수적**. `python3 -c "open('x.py').read()"`(workdir 내 정상 읽기)도 exit 3로
  잡힘 — 로그 텍스트만으론 workdir 내/밖 확신 불가하므로 안전측. **오탐이 아니라 fail-closed.**
- 대가: 인터프리터 파일 I/O가 잦은 워크플로면 P1 사람 검토가 자주 뜸. 이건 "감사기가 완벽 차단막이
  아님"의 정직한 반영이지 결함이 아님. 격리가 서면 이 플래그 의존도가 낮아짐.

## 결론 (브라이언과 동일)
- 로그 감사기 위상 = **탐지 보조 + 사각지대 표면화**. "완전한 차단막" 아님.
- 진짜 경계 = (a) P1 사람 diff/로그 검토 + (b) 컨테이너 네트워크/FS 격리.
- **수장 결정 항목(컨테이너 격리)을 QA와 함께 최우선으로 올림.** 격리가 있으면 감사기 미탐과
  무관하게 유출이 물리 차단. 3겹 방어(사전스캔+로그감사+diff/QA)는 격리 전까지의 실질 방어선.
- FN4의 "완벽 탐지"는 목표 아님(불가). opaque 표면화로 과신만 차단하고, 근본은 격리로.
