01

한 장 요약

PM 작성

이 플러그인이 무엇이고, 누가 왜 쓰는지 — 비개발자도 이해할 수 있는 한 문단.

이 도구는 클로드 코드에 한 번만 설치해두면, 새 프로그램을 만들 때마다 설계 → 기록 → 코딩 → 검사의 네 단계를 자동으로 밟아주는 플러그인입니다. 먼저 클로드가 면접관처럼 질문을 던져 "무엇을 만들지"를 빈틈없이 확정하고, 그 결정을 문서로 남긴 뒤, 실제 코드 작성은 코덱스(다른 AI)에게 맡깁니다. 마지막으로 그 코드가 제대로 됐는지 세 관문으로 검사해서, 통과한 것만 "완료"로 인정합니다.

무엇을

기획·기록·코딩·검사를 한 번에 굴리는 클로드 코드 플러그인.

누가 쓰나

아이디어는 있는데 구현 디테일과 검증이 늘 빠지는 팀 또는 1인 개발자.

왜 쓰나

AI가 "다 만들었다"고 말하는 걸 그대로 믿지 않고, 실제 증거로 검사해 완료를 판정하기 때문. 위험한 순간엔 반드시 사람이 확인.

02

전체 흐름 — 4단계 파이프라인

흐름도: 엘레나

한 번의 지시로 아래 네 단계가 순서대로 흐릅니다. 노란 태그는 자동 게이트, 빨간 태그는 사람 승인이 반드시 걸리는 지점입니다.

1

설계

grill me

클로드가 면접관이 되어 기획의 빈틈을 캐묻고 요구사항을 확정. 쉬운 80%는 자동, 진짜 갈리는 것만 사용자에게.

사람: 갈리는 결정(P4)
2

ADR 확인

grill-with-docs

결정을 ADR로 기록하고 용어집(CONTEXT.md)과 대조. 확정된 결정은 문서에 인라인 갱신.

문서 정합성 체크
3

코덱스 코딩

codex mcp-server

Claude가 오케스트레이터로 Codex를 호출. 격리 작업공간에서 코딩하고 결과를 diff로 회수.

사람: apply 승인(P1)
4

검사

3단계 게이트 (G1·G2·G3)

문법·경계 → 실제 동작 → 교차검토. 세 관문을 증거로 다 통과해야 "완료".

완료 = G1 ∧ G2 ∧ G3
◆ 핵심 원칙 — 상태는 전부 파일로, 승인은 게이트로, AI의 자기보고는 믿지 않고 교차검증. 코딩(Codex)과 검토(Claude)는 반드시 다른 벤더가 맡습니다.
03

각 단계 상세

네 단계를 각각 카드로 풀어봅니다. 3-1·3-2는 설계/DX, 3-3은 개발, 3-4는 QA 소유입니다.

3-1 · 설계 (grill me) 엘레나 DX

클로드가 요구사항을 압박 테스트

객관식·한 번에 하나씩·2~4선택지 추천형으로 결정 피로를 없앱니다(힉의 법칙). 원본 grill me 위에 우리가 반드시 얹는 4가지:

  • 인지부하 분배 — 쉬운 80%는 AI끼리 자동 해소([CONSENSUS]), 진짜 판단만 [NEEDS MAX]로 사용자에게
  • 진행감 표시 — "5개 차원 중 3/5 해소" 식 피드백(가시성 원칙)
  • "왜 묻는지" 한 줄 — 갈리면 아키텍처가 바뀌는 이유를 곁들여 답의 질을 높임
  • 핸드오프 — 결정 요약이 死문서로 끝나지 않고 그대로 3-2 ADR·스펙의 입력으로 흐름

3-2 · ADR 확인 설계 골자

결정을 기록하고 대조

grill me가 확정한 결정을 근거와 함께 남기고, 다음 단계가 그 위에서 일하게 합니다.

  • 결정 기록 — 각 결정을 "무엇을 / 왜 / 대안은" 형태의 ADR로
  • 용어집 대조 — CONTEXT.md 도메인 용어와 충돌·누락 점검
  • 인라인 갱신 — 결정이 나면 관련 문서를 그 자리에서 갱신(문서-현실 드리프트 차단)
  • 영상의 교훈 — "grill me는 비전만 풍부, 구현 디테일이 빔" → 이 단계에서 유저플로·엣지케이스·보안을 보강

3-3 · 코덱스 코딩 (MCP) 백팀장

Claude 오케스트레이터 → Codex 실행

확정된 사실: codex mcp-server 주력 + codex exec 보조. 백팀장이 실물 래퍼로 검증했습니다.

  • 호출 — .mcp.json이 래퍼(codex-mcp.sh)를 경유해 codex mcp-server 기동. 노출 툴은 codex·codex-reply 2개(approval-policy·sandbox 인자로 정책 제어)
  • 흐름 — 일회용 격리 workdir 생성 → Codex가 코딩 → 결과를 diff로 회수(사람 승인 자료 P1)
  • 편입 — codex_run.sh가 workdir 생성·위임·diff 캡처를 담당, 이어서 qa_verify.sh로 자동 검사 연결

3-4 · 검사 브라이언 QA

완료는 선언이 아니라 게이트가 증거로 판정

결과물은 G1→G2→G3 순서로 통과해야 "완료". 싼 검사 먼저, 비싼 교차검토 마지막. 하나라도 미통과 = 재작업. 상세 관문은 아래 세로 흐름 참고.

  • G1 문법·경계 — 컴파일 성공 + 변경이 격리 안 + 요구 파일 생성
  • G2 실제 동작 — 테스트 진짜 실행 + 회귀 0 + 요구사항 매핑
  • G3 교차검토 — 다른 AI 검토 + 결함 주입 생존성 + 보안 스캔

04

플러그인 구조

백팀장 확정

자체완결 디렉토리 + 매니페스트. 아래는 백팀장이 실제로 만들어 검증한 스켈레톤입니다(JSON 3종 유효·훅 차단·회귀 10/10 실측 통과).

plugin/ ├─ .claude-plugin/ │ └─ plugin.json # name·description·version (JSON 유효 ✓) ├─ .mcp.json # codex를 MCP로 등록 — 래퍼 경유 ├─ skills/ │ └─ harness-run/SKILL.md # 4단계 오케스트레이터 지침 ├─ hooks/ │ ├─ hooks.json # PreToolUse: Bash 위험명령 차단 │ └─ guard_dangerous.sh # exit 2 = 차단 (P2·P3) ├─ bin/ │ ├─ codex-mcp.sh # MCP 래퍼 (CODEX_HOME 자동세팅, 권한trap 우회) │ ├─ codex_bootstrap.sh # CODEX_HOME 표준화 │ ├─ codex_run.sh # Codex 호출 + diff 캡처(P1 자료) │ ├─ qa_verify.sh # G1·G2 자동화 (브라이언 실물) │ └─ scan_danger.py # 위험패턴 스캐너 ├─ lib/ │ └─ danger_patterns.txt # 위험패턴 데이터 분리(6분류) └─ tests/ └─ test_scan_danger.py # 회귀 고정 케이스 (10/10 통과)
◆ 구조 규칙 — commands/agents/skills/hooks는 .claude-plugin/ 안에 넣지 않고 루트에 둡니다. 테스트는 claude --plugin-dir ./plugin으로 설치 없이 로드.
✓ MCP 호출 방식(실측 확정) — .mcp.json이 codex mcp-server를 직접 부르지 않고 래퍼(codex-mcp.sh)를 경유합니다. 래퍼가 CODEX_HOME을 자동 세팅해 root:600 설정파일 문제를 우회하고, ${CLAUDE_PLUGIN_ROOT} 치환으로 설치 위치와 무관하게 동작. 실측: 래퍼 기동 → initialize 응답 codex-mcp-server 확인.
05

안전장치

브라이언 QA

이 서버는 codex 내장 샌드박스를 물리적으로 못 켜므로, 4겹 경계로 대체합니다. 그 위에 위험 코드 차단과 사람 승인 지점을 명문화합니다.

먼저 오해부터 풉니다 — claude의 안전장치는 codex에 자동으로 넘어가지 않습니다

수장님 질문: "claude에 채운 hook·skill이 codex 코딩에도 적용되나?" → 답은 아니오. claude와 codex는 완전히 다른 일꾼(별도 프로세스)입니다. claude에 채운 안전벨트는 codex에는 채워지지 않습니다.

claude (오케스트레이터)
설계·검수를 맡는 일꾼
  • 훅이 걸림위험명령 실행 직전 차단(PreToolUse=Bash)
  • 스킬을 읽고 따름design-grill·adr-check 지침 적용
≠
codex (코딩 일꾼)
격리 방에서 코딩만 하는 일꾼
  • claude 훅 안 걸림codex는 claude 툴이 아니라서
  • claude 스킬 안 읽음별도 프로세스라 지침이 안 닿음
  • codex 자체 안전장치도 OFFapproval=never · sandbox=full-access (우리가 끔)

그럼 codex는 무법지대? 아닙니다. 통제 방식이 다릅니다 — "실행 순간 차단"이 아니라 "격리 방에 가두고 결과물을 검문"하는 모델입니다.

⚠ 이 모델의 근본 구멍 — 사후 diff는 "파일이 어떻게 바뀌었나"만 남깁니다. codex가 코딩 도중에 이미 낸 부작용(격리 방 밖 파일 삭제, 외부 네트워크 호출)은 diff에 안 남아 사후에 못 잡을 수 있습니다. "실행 중 위험을 어떻게 막을지"가 이 설계의 핵심 미결 과제입니다.
✓ 실측 결론 — 구멍은 로그로 메운다 · codex는 실행한 모든 셸 명령을 .codex_home/sessions/**/rollout-*.jsonl에 기록한다(PM 실측: 로그에서 printf > /tmp/... 같은 방 밖 쓰기 기록까지 추출됨). diff는 "무엇이 남았나", 로그는 "무엇을 했나" — 둘 다 봐야 완전하다.
겹방어무엇을 막나
(a) 사전위임 프롬프트 실행 전 스캔위험 지시가 담긴 코딩 요청 자체를 거부
(b) 실행 중 감사 ★qa_verify에 로그 스캔(L1.5) 추가 — rollout 로그의 exec_command를 위험패턴과 대조diff에 안 남는 실행 중 행위(방 밖 삭제·외부 네트워크·유출) 사후 포착. 이미 있는 데이터라 비용 낮음
(c) 격리 강화가능 시 네트워크 차단 실행cwd 고정으로 못 막는 절대경로 쓰기·외부 접속을 환경 차원에서 차단

실측 근거: 브라이언 — 부작용 3종(임시파일 생성후삭제·방 밖 읽기·외부 네트워크) 전부 diff 흔적 0 실증 + rollout 로그에 exec_command 남음 발견. PM — 로그 15개에서 실제 실행 명령(방 밖 /tmp 쓰기 포함) 추출 성공. → (b) 로그 스캔이 실행 중 사각지대를 메우는 핵심, Phase 2 최우선.

사람 승인이 반드시 필요한 지점

#승인 지점왜 사람이 필요한가기본 동작
P1코드베이스로 apply 전격리 결과를 실제 프로젝트에 반영하는 순간이 진짜 위험diff 제시 → 승인 대기 (자동 apply 금지)
P2위험 패턴 무시하고 강행스캐너가 잡았는데 강행하려면 사람이 근거 대고 판단--strict 차단 → 명시 해제해야
P3위험 플래그 사용샌드박스/승인 우회는 그 자체가 고위험 결정수장 1회 승인 경계 안에서만, 확대는 재승인
P4설계상 진짜 갈리는 결정근본적으로 갈리는 요구는 AI가 추측 금지자동 해소 불가분만 사용자에게 질문
P5게이트 2회 재작업 후 미통과무한 루프 방지, 사람이 방향 재설정2회 실패 → 중단·에스컬레이션
⚠ 절대 자동화 금지 — P1(apply)과 P2(위험 무시)는 어떤 경우에도 사람 승인 없이 통과시키지 않습니다. 나머지는 정책으로 조정 가능하되 기본은 안전측(fail-closed).

설치·사용법

PM 작성

한 번 설치한 뒤 실제로 어떻게 쓰는지.

  1. 설치 — 로컬 검증은 claude --plugin-dir ./plugin으로 설치 없이 로드하고 /reload-plugins로 갱신. 배포 후에는 마켓플레이스 /plugin install.
  2. 선결 1회 세팅 — codex_bootstrap.sh로 CODEX_HOME을 표준화(root:600 설정파일 문제 우회). 이 서버는 코덱스 내장 샌드박스를 못 켜므로 "컨테이너 격리 + 일회용 workdir + diff 승인"을 안전 경계로 사용.
  3. 사용 — 한 줄 지시로 4단계 파이프라인이 순서대로 흐름. 사람은 P1(코드 반영 승인)과 P4(정말 갈리는 결정)에서만 개입. 나머지는 게이트가 자동 판정.
  4. 결과 — 코덱스가 격리 공간에서 코딩한 diff를 사람이 확인하고 승인하면 실제 프로젝트에 반영. 검사(G1·G2·G3) 통과 못하면 자동 재작업.
◆ 미해결 1건(환경) — claude --plugin-dir 통합 로드는 이 세션의 claude CLI 인증(Not logged in)이 풀려야 최종 확인 가능. 개별 컴포넌트는 전부 실측 통과했으므로 코드 결함이 아닌 인증 대기 항목.
