Opening — Harness Multi-Agent 2.0 소개
클로드·코덱스·제미나이를 동시에 사용하는 하네스 멀티에이전트의 업그레이드 끝판왕 영상. 이전 영상(하네스 멀티에이전트 시스템)에 대한 피드백을 반영해 2.0 제작. 처음 오는 사람도 이해 가능하게 제작. 이전 버전: 클로드 코드의 Opus가 오케스트레이터가 되어 코덱스·제미나이 같은 다른 AI들을 워커로 함께 작업하는 구조. 그런데 여기에 4가지 문제가 있었음.

기존 시스템의 4가지 문제
1. 왜 클로드 코드가 오케스트레이터가 돼야 하는가? — 클로드 코드 하나만 쓸 때는 전적으로 신뢰하게 되지만, 여러 모델을 동시에 쓰며 서로 비판하게 하면 클로드 코드의 약점이 드러남. 딥시크(?) 최근 벤치마크에서 GPT 5.5가 Opus를 압도. 앤트로픽이 요금을 올리거나 종량제로 바꾸면? 하네스 멀티에이전트 시스템은 벤더에 종속되면 안 됨.
2. 비용 — 모델을 여러 개 쓰면 품질↑, 여러 모델 아니면 못 하는 작업도 있음. 그러나 비용 문제. 기본 전제는 클로드코드 기본 구독 + 코덱스 기본 구독. 여기에 제미나이 구독까지 붙으면 비용 과다. 다행히 Antigravity CLI를 통해 Gemini 최고 모델을 무료로 쓸 수 있어 해결. 클로드 코드·코덱스 두 구독도 부담이면 모델을 조절 가능(예: 클로드 코드만 구독 + 제미나이 무료 조합).
3. 벤더 독립성(가장 중요) — 만든 시스템이 클로드 코드/코덱스/Antigravity 어디에서도 돌아가는가? AI 기술은 급변, 어제 1등이 오늘 2등. 벤더가 고객에 반하는 정책을 펼 수도. 하네스 멀티에이전트 시스템은 어느 벤더에서나 자유롭게 작동해야 함. 하네스 멀티에이전트 2.0은 클로드 코드에서 켜면 Opus가 오케스트레이터, 코덱스에서 켜면 GPT 5.5가 오케스트레이터, Antigravity에서 켜면 Gemini 3.1(Pro/High)이 오케스트레이터가 되도록 설계.
4. 설치 과정 — 이전 버전은 폴더에 필요한 파일을 준비하는 방식이라 환경 변화에 유연하지 않고 3번 이유 때문에 한계. 2.0에서는 클로드 코드/코덱스/Antigravity 어디든 "플러그인 설치 → 멀티에이전트 시스템 구성해줘" 한 줄이면 끝. 플러그인을 한 번만 설치하면 이후 어떤 폴더에서든 사용 가능. 플러그인이 내 작업 폴더에 자동으로 필요한 파일들을 준비.

이 영상으로 얻는 것: (1) 실제 작동하는 하네스 멀티에이전트 시스템 2.0, (2) 작동 원리, (3) 매뉴얼. 순서: 설치 및 시연 → 작동 원리.

기본 용어 — Harness, Orchestrator, Worker
- 하네스(Harness): 원래 말을 채우는 고삐와 안장(제어 장치). 코덱스·제미나이 같은 AI는 힘은 세지만 제멋대로인 말. 멀티에이전트 시스템 없이 그냥 불러내면 제어가 안 됨. 이 강력한 말들을 고삐로 단단히 묶어 통제하는 것이 하네스 엔지니어링, 그 위에 세운 것이 하네스 멀티에이전트 시스템.
- 오케스트레이터(지휘자): 전체 작업을 효율적으로 쪼개고, 어떤 워커에게 일을 시킬지 정하고, 최종 결과를 하나로 합침.
- 워커(실무 AI): 지휘자가 시킨 일을 실제 수행하는 클로드·코덱스·제미나이 등. 지휘자 1명이 여러 실무 AI를 부림.
- 팬아웃(Fan-out)/팬인(Fan-in): 하나의 작업을 여러 워커에게 동시에 펼쳐 맡기는 게 팬아웃, 그 결과물들을 지휘자가 다시 하나로 모으는 게 팬인.
- 워커 호출 방식 3가지: (a) 내장 호출 — 같은 벤더 내에서 워커 호출(예: 클로드 코드에서 Opus 오케스트레이터가 내부적으로 서브에이전트를 호출해 클로드 워커 준비). (b) CLI 호출 — 오케스트레이터가 외부 워커를 CLI 터미널 명령어로 부름(예: 클로드 코드에서 제미나이 호출). (c) MCP 호출 — MCP 규격으로 외부 워커 호출(MCP = AI용 USB 포트, 외부 도구 연결 표준). 어떤 방식이 되는지는 환경마다 다르며 멀티에이전트 시스템이 알아서 처리.
- 핵심: 지휘자는 어느 벤더에서 켜느냐에 따라 바뀜(클로드→클로드, 코덱스→GPT, Antigravity→제미나이). 지휘자는 바뀌어도 시스템은 똑같음.

시연 1 — 클로드 코드에서 MAT 기능 업그레이드
서로 다른 세 작업을 서로 다른 세 환경에서 실행. 설치는 "플러그인 깔고 멀티에이전트 시스템 구성" 한 줄. MAT(멀티에이전트 트래킹 프로그램) = 멀티에이전트 시스템은 폴더와 파일로만 구성되므로, 실행 중 상태 정보·로그를 확인할 별도 프로그램이 필요해서 이 시스템으로 만든 첫 프로그램. MAT은 워커들의 호출 상태를 보여줌. 그런데 오케스트레이터가 각 워커를 어떤 방식(CLI/MCP/API)으로 호출하는지 정보가 없음 → 그 정보를 표시하는 기능을 추가하는 작업.
과정: 클로드 코드에서 플러그인 명령 실행 → 마켓플레이스 등록(정식 버전은 GitHub, 영상은 로컬 폴더 경로) → 사용 환경에 따라 설치 범위 결정(현재 폴더에서만 설치) → reload plugins 명령 → "멀티에이전트 시스템 구성해줘" 요청 → 플러그인 스킬이 필요한 파일을 현재 폴더에 복사 → 오케스트레이터 결정(클로드 코드니 클로드 선택) → 작업 폴더 선택 → 설치 상태 점검(모두 PASS) → 폴더 안 파일 확인.
기능 추가 전에 클로드·코덱스·제미나이 세 모델에게 동시에 의견 요청(팬아웃/팬인/병렬 패턴). 기능 추가 가능, 유일한 이견은 "모델 호출 방식을 화면 어디에 표시할지". 클로드 메인이 기능 추가 → 코덱스 크리틱이 검수 → 버그 2개 발견 → 클로드 메인이 워커 2개 병렬로 수정 → 완료. MAT에서 각 워커 호출 방식 확인 가능.

시연 2 — 코덱스 앱에서 PDF→HWPX 한글 문서 변환
코덱스 앱에서 실행(오케스트레이터 = GPT 5.5). 플러그인 설치(왼쪽 메뉴 → 플러그인 → 멀티에이전트 마켓플레이스 추가 → 플러그인 추가) → 대화창에서 "멀티에이전트 시스템 구성" 요청 → 전부 PASS → 파일·폴더 확인.
작업: 수식 가득한 PDF 한 페이지를 Gemini 비전이 읽어 분석하고, 코덱스가 그걸 한글 HWPX 문서로 수식까지 원본과 똑같이 재현(지난 영상에서 클로드 코드로 보여준 작업). MAT에서 진행 모니터링 — Gemini가 PDF 분석 → 완료 → 코덱스가 그 결과물 파일 + HWPX MCP로 최종 한글 파일 생성 → 완료. 수식 대부분 일치, 조립제법 부분만 미변환(한글 수식에 없거나 Gemini가 해당 수식 문법 모름). 문서 작성에 다양하게 활용 가능.

시연 3 — Antigravity에서 AI 의견 종합
Antigravity에서 실행(오케스트레이터 = Gemini). 플러그인 설치(정식은 명령어 입력) → "멀티에이전트 시스템 구성" 요청 → 오케스트레이터/설치 폴더 물어봄(기본으로) → 설치 완료.
작업: "앞으로 10년 AI는 어디로 가는가"에 대해 클로드·코덱스 워커가 서로의 답을 보지 않고 각자 의견(팬아웃) → 지휘자 Gemini가 자기 관점까지 더해 하나로 종합(팬인). MAT에서 클로드·코덱스 호출 확인(앞서 수정한 기능 덕분에 호출 방식도 확인 가능). 코덱스 먼저 완료 → 클로드 완료 → 종합. (이 정도 작업은 굳이 멀티에이전트 필요 없지만 시연용 간단 예시.)

작동 원리 — 같은 시스템, 바뀌는 오케스트레이터
핵심 원리는 하네스 멀티에이전트 1.0과 동일: 상태를 파일로 저장하고, 승인 받고, 검증하고, 컨텍스트를 관리. 여기에 개선 사항이 추가된 것.
목적: 각 모델의 장점을 최대한 살려 최상의 결과물 도출. 모델마다 잘하는 게 다름(코딩, 검수, 비전, 긴 문서 분석). 한 모델에만 맡기면 그 모델의 약점까지 결과물에 반영됨. 그래서 여러 모델이 서로 만들고 서로 비판하게 함.
구조: 지휘자(오케스트레이터) 1명 + 그 아래 워커들. 오케스트레이터가 작업을 쪼개고 누구에게 시킬지 정하고 결과를 합침. 워커는 실무 AI(서브에이전트 같은 역할).
핵심: 이 지휘자가 특정 벤더에 고정돼 있지 않음(이번 버전의 핵심). 같은 시스템, 지휘자만 바뀜. 어느 벤더에서 켜느냐에 따라 지휘자 변경(클로드 코드→Opus, 코덱스→GPT 5.5, Antigravity→Gemini 3.1 Pro/High). 지휘자가 바뀌면 워커 구성도 맞게 바뀌지만, 같은 플러그인·같은 폴더 구조·같은 작업 명령. 바뀌는 건 지휘자뿐. 이게 중요한 이유: 어느 한 벤더가 정책을 바꾸거나 가격을 올려도 작업은 다른 벤더에서 그대로 돌아감.

워커 구성 및 MCP/CLI 호출 방식
각 워커의 역할:
- 클로드 메인 & 코덱스 메인: 메인/보조 코딩과 설계, 분석, 테스트, 이미지 생성 담당. 이 둘은 체감 성능이 거의 비슷해 메인을 바꿔도 무방.
- 크리틱(검수) 워커: 다른 워커의 산출물을 공격적으로 비판·검수. 테스트해 보면 검수 워커가 평균 4~5개 문제를 찾고 원 작업 워커가 그중 2~3개를 수정.
- 제미나이 워커: 비전 능력, 긴 문서 분석, 제3자 시각 검토 담당.
- 원칙: 검수는 항상 지휘자와 다른 벤더가 맡음(자기가 만든 걸 자기가 보지 않게). 셋 다 LLM이지만 계보가 다르면 놓치는 지점도 다름. 작업에 직접 손대지 않은 바깥 모델이 안에서 굳어진 가정을 다른 눈으로 한 번 더 걸러줌. 매 작업마다 모든 워커를 부르지는 않음 — 필요한 워커만 사용자 승인 하에 부름. 워커 구성은 벤더에 따라 조금씩 다름.
- 벤더가 다른데 같은 시스템이 도는 방법: 워커 호출 방식(내장/MCP/CLI/API)의 차이를 backend.json 파일 한 곳에 모으고, 디스패처(dispatcher)가 그 워커에 맞는 방식으로 알아서 호출. 덕분에 각 환경에서 최적 연결 방식을 쓰면서도 시스템은 똑같이 동작. Gemini는 Antigravity CLI를 통해 최고 모델을 무료로 사용.

컨텍스트 엔지니어링 원칙 3가지
1. 컨텍스트 비대화 방지 — 컨텍스트가 방대해지면 AI가 중간에 묻힌 정보를 놓치고 품질 하락. 작업 진행 내용은 context.md에 1500자 제한, 워커 지시는 brief.md에 1200자 제한으로 짧게 정리. 초과되면 핵심만 남기고 나머지는 로그로 넘긴 뒤 비움. 긴 문서·코드는 컨텍스트에 붙이지 않고 소스 폴더에 파일로 두고 경로로만 워커에게 전달.
2. 다 미리 펼치지 않고 필요할 때 필요한 만큼만 꺼냄 — 책상의 모든 서류를 쏟는 게 아니라 필요한 철 하나만 꺼내 봄.
3. 워커 결과는 바로 다음 단계로 넘기지 않고 다른 워커가 검수한 뒤 통과시킴.
추가 팁: 워커에게 보내는 brief를 쓸 때 가장 중요한 목표와 제약은 앞쪽에, "하지 말 것"은 끝쪽에 둠(모델은 글 가운데 있는 걸 잘 놓침).
실행 흐름(모든 작업이 같은 길): 사용자가 자연어로 작업 요청 → 오케스트레이터가 자동으로 task.md 작성 → 워커 호출에 대한 사용자 승인 요청 → 승인되면 각 워커에게 지시할 내용을 brief.md에 정리 → 워커 실행 → 오케스트레이터가 각 워커의 응답을 result.md로 저장 → 크리틱 워커가 검수. 전체 실행 흐름은 로그 파일에 기록. 모든 상태가 파일로 남기 때문에 세션이 끊겨도, 작업자가 바뀌어도 파일만 읽으면 이어짐. 검수에 실패하면 다음 단계로 안 가고 그 워커를 다시 부름.

가드레일과 자기 개선 구조
- 가드레일(통제): 어떤 모델이든 부르기 전에 사용자가 명시적으로 승인하게 함. 이유: 예전에 코덱스에 긴 번역 작업을 시켰는데 끝나고 보니 클로드 코드가 사용량 제한에 걸려 있었음 — 코덱스가 허가 없이 클로드 코드를 호출한 것. 그래서 내가 모르는 호출은 일어나지 않도록 승인 강제. 워커 결과물은 검증돼야 다음으로 넘어가고, 모든 작업 흐름은 로그로 남겨 추적 가능.
- 자기 개선: 매 작업이 끝나면 오케스트레이터가 다음에도 다시 쓸 교훈만 추려 learnings.md에 기록. 새 작업 시도 시 그 파일을 먼저 참고해 같은 실수 반복 방지.
- 시스템 자체 수정: 내용이 간단하지 않아 매번 AI에게 처음부터 설명하며 고칠 수 없음. 그래서 두 파일을 둠 — design_basis.md(이 시스템을 왜 이렇게 만들었는지), system_invariants.md(수정할 때 반드시 지켜야 할 규칙). 이 파일들은 평소엔 읽지 않고 시스템을 설명하거나 고칠 때만 참고. 덕분에 매번 처음부터 설명하지 않아도 일관되게 안전한 수정 가능.
정리: 핵심 원리는 지난 버전과 같음(상태를 파일로 남기고, 승인 받고, 검증하고, 컨텍스트를 아낌). 이번 버전은 "어느 벤더에서도 그대로 돈다"를 더함 + 몇 개 개선 사항 추가. 이게 하네스 멀티에이전트 2.0.

Ending — 벤더 독립적 AI 협업 시스템
어느 벤더든, 어떤 모델이든, 어떤 호출 방식이든 구애받지 않고 자유롭게 제어할 수 있는 하네스 멀티에이전트 시스템이 필요했음. 특정 벤더가 갑자기 불리한 정책을 취하든, 주로 쓰는 AI 모델을 바꾸고 싶든, Antigravity의 제미나이가 갑자기 유료화되든, 특정 기업에 종속되지 않는 독립 시스템이 필요했기 때문. 하네스 멀티에이전트 시스템은 작업이 복잡할수록, 최상의 결과물이 필요할수록, 특정 모델의 한계가 느껴져 AI들의 공동 작업이 필요할 때 진가를 발휘. 매뉴얼은 고정 댓글의 구글 폼에 이메일 남기면 발송.
