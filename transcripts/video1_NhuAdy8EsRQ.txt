인트로 — 무료 공개 ③ SaaS MVP를 기획부터 자율 구현까지
네, 안녕하세요. 실벨 개발제입니다. 제 클로드코드 에전틱 엔지니어링 강의 일부를 무료로 공개하는 세 편짜리 시리즈의 마지막 세 번째 영상입니다. 이번 영상은 실제로 이제 사스 MVP 하나를 처음부터 만들어 보는 실습인데요. 먼저 유명한 스킬이죠. 그릴미 스킬로 AI를 면접관처럼 세워 가지고 제가 짠 기획의 빈틈을 이렇게 집요하게 캐묻게 할 거예요. 그렇게 단단해진 기획 문서를 하네스 프레임크를 이용을 해서 페이즈와 스텝으로 잘게 쪼개고 루프 엔지니어링으로 자율 실행까지 시켜서 실제로 이제 동작하는 기능으로 구현하는 것까지 한번 다뤄보도록 하겠습니다.

Part 1. Grill me 스킬 — AI 면접관 설치 (Matt Potka, GitHub 15만 스타)
GR me(Grill me) 스킬을 구글에서 검색해 찾음. 만든 사람은 Potka. 스킬 링크를 복사해 VS Code에서 새 터미널 열고 클로드 코드 실행 → checkout main으로 PRD/ADR 스캐폴딩 초기화. 그릴미 스킬 설치. 스킬 설명(한국어 번역): "공유된 이해에 도달할 때까지 계획이나 설계에 대해 사용자를 집요하게 인터뷰하고 결정의 틀(각 갈래)을 하나씩 해소한다. 계획을 압박 테스트하거나 설계를 추궁받고 싶을 때 사용." 이 스킬은 로직이나 스크립트가 전혀 없는 순수 프롬프트 한 가지. GitHub 스타 11만6천개(영상 내 언급, 화면상 매트 포트카 리포).

핀테크 SaaS 기획 시작 → AI가 보안·프라이버시를 집요하게 캐묻기 시작
Grill me 스킬 사용. 요구사항: "핀테크 SaaS 앱을 만들 거야. CSV 파일(카드 명세서)을 업로드하면 그 정보를 받아서 Claude API(Opus 4.8)로 AI 분석·인사이트 제공. 랜딩 페이지 + 구독 결제 + 구독자에게 더 좋은 피처. 랜딩→대시보드 이동." AI가 코드베이스 현재 상태 확인 후 인터뷰 시작.
- 가장 무거운 질문: 금융 PII를 웹으로 보낸다. 무엇을 저장하고 무엇을 Anthropic 서버로 전송하느냐가 DB 선택/보안 요구사항/법적 책임을 전부 결정. Anthropic은 데이터를 학습에 쓰지 않고 입출력 일정 기간 보관 후 삭제하지만, 외부 서버로 카드 거래 내역이 나간다는 사실은 변하지 않으므로 보내기 전 마스킹/제거·암호화가 중요한 설계 분기.
- 업로드한 CSV를 서버에 저장할까 분석 후 버릴까? → 저장해야 컨텍스트 기반 개인화 제공 가능. 보안 잘 설계 필요. DB 필수, 삭제 정책, 유출 시 대응 책임. 시계열 트렌드·월별 비교·과거 명세서 기능이 핵심 가치.
- 저장은 DB 얘기, Claude API로 보낼 때는 마스킹 후 전송.
- DB/인증 조합: Supabase로 감. 1차 타겟 시장: 한국 우선. 국내 결제: 폴라(Polar) 사용.
- 은행 카드마다 다른 CSV 포맷 → Claude가 컬럼 자동 매핑/파싱.
- (8번째 질문) AI 분석 인사이트가 구체적으로 무엇을 만들어 내는가가 제일 중요. 결과물 형태: 구조 리포트 우선 추천.
- (9번째) 수익화 핵심: 프로 플랜이 뭐냐. 무료 가치를 충분히 맛보게 하고 업그레이드할 명확한 이유 필요. 주된 게이트는 사용량. 무료는 월별 리포트 제한, 유료는 별개 기능. 추천으로 감.
- 비용 현실: 프리미엄대라서 무료 사용자 명세서까지 전부 Opus로 분석하면 비용 큼. 모델 라우팅(티어 라우팅, 10배 선처리)이 자연스럽게 무료 게이트와 맞물림.
- 온보딩 흐름: 랜딩→대시보드. 샘플 데이터 데모 먼저 보여주고 그 다음 가입.
- 빌드 순서: 핵심 분석 루프 먼저.

12개 질문으로 결정 트리 완주 — 결정 피로를 없애는 추천형 인터뷰
결정 트리(decision tree)를 끝까지 내려옴(약 12개 질문). 공유된 이해를 한 장으로 정리: 한 줄 정의, 데이터, 프라이버시, 티어, 온보딩, 빌드 순서, 아직 미정 항목(추적: 가격 등). 그릴미 인터뷰 자체는 매우 좋으나 아키텍처/보안은 추가 검토 필요. 초안 작성엔 좋음. 발표자 소감: 그릴미(기획/PRD 작성)와 본인이 기존에 하던 순서(유저플로→아키텍처→엣지케이스·에러핸들링·보안→오버엔지니어링 점검)를 합치면 좋겠다. PRD 생성엔 좋지만 실질적 아키텍처·ADR로 옮길 땐 다른 스킬 필요.

Grill me 결과 vs 수동 설계 비교 — 비전은 풍부, 구현 디테일은 빈다
main 브랜치(수동 설계 30~40분 소요) vs 그릴미 브랜치 문서 비교. 차이: 모델 전략(듀얼 모델 vs 단일), 컬럼 매핑, 마스킹(데이터 집계 통계만 vs 원본 거래내역 암호화 저장), 게이팅 응답 규약(미명시 vs 명시), 외부 API 래퍼(느슨 vs 상세), TDD 언급 유무, 테이블 추상 vs 상세(7개 테이블), 문서 성격(구현 명세 중심). 결론: 수동 설계 쪽이 구현 준비도가 명백히 높다. 그릴미 산출물은 비전/내러티브가 더 풍부하지만 구현 디테일은 비어 있다. → 그릴미로 구현 비전 잡고 이후 구현 디테일을 애드. 두 문서의 좋은 것들을 병합.

충돌 결정 해소 → 문서 병합 → 최종 PRD 확정
충돌되는 결정(룰베이스냐 자동이냐, 표준 컬럼 포맷, 마스킹된 거래단 vs 집계 통계 등)을 리졸브. 글로벌 영어 우선 vs 한국 우선 → 글로벌로 감. 룰베이스 통계, 표준 파서 + Claude 폴백 하이브리드. Claude 보내는 데이터 경계(프라이버시/토큰비용/인사이트 해상도 트레이드오프) → 마스킹. 그릴미 강점 재확인: 합의까지 집요하게 나감, 결정 트리를 스스로 생성해 결정 항목을 트리로 구조화(건너뛰지 않음), 한 번에 하나씩 대화형으로 빠르게 결정, 추천 답 제공으로 인지 부하·결정 피로 최소화. 문서 병합 → 한 번 더 검토·simplify → 클로드 디자인으로 넘어가 DESIGN.md 작성 예정. main에 머지.

Part 2. 하네스 장착 — 구현은 Codex로 위임 (계획=Claude · 코딩=Codex)
구현은 Codex로. executor.py 스크립트가 지금은 claude -p (헤드리스 모드)로 돌리고 있는데, 이를 "코덱스화(codex exec 헤드리스 모드)"로 전환. 새 세션에서 "claude로 되어 있는 걸 codex 시켜줘"라고 하고 스크립트를 넣음. 이유: 계획 단계는 전부 Claude로, 코딩은 Codex(GPT 5.5)에게 위임. 두 모델을 적재적소에 번갈아 사용하는 것도 개발자 역량.
발표자 관점(중요): 스펙을 작성하고 그 스펙으로 롱러닝 에이전트(사람 간섭 없이 자율적으로 오래 일하는 에이전트)를 돌리려면 좋은 하네스가 필요. 근데 이 하네스도 모델이 만드는 시대가 오고 있다. 클로드 코드 자체가 이미 하네스이고, 하네스가 하네스를 만든다(메타 하네스). 클로드 코드에게 "이런 하네스 만들어 줘" 하면 잘 만들어 줌. 그럼에도 우리 역할은 하네스를 이해하는 것 — 하네스는 결국 시스템 설계이므로, 이해하고 쓰는 것과 이해 못하고 위임하는 것은 다르다. 에이전트 엔지니어링/하네스 엔지니어링의 원리를 알아야 설 자리가 넓어진다.
코덱스화 후 변화: claude 호출 → codex exec 호출로 바뀜. 컨벤션 파일도 CLAUDE.md → AGENTS.md 읽는 것으로 변경. AGENTS.md 생성. codex 설치되어 있어야 가능. 알아서 테스트 실행해서 검증 → 이게 검증 루프(Plan·Execute·Verify 루프). 통과하면 다음 단계로. AGENTS.md 생성 → Plan → Exec → Verify 루프 반복. 하네스 시작 전 컨텍스트 초기화 권장.

Codex 훅 별도 설정 필요
코덱스는 훅을 따로 셋업해야 함. Claude는 .claude 안에 스킬·훅이 셋업돼 있지만 Codex는 없으므로 코덱스 훅 셋업 필요. TDD 가드가 codex exec 할 때 트리거돼야 하는데 코덱스 훅 없으면 트리거 안 됨. AGENTS.md에 명시 지침을 해도, "지침"과 "강제성 있는 시스템"은 다름 — 둘 다 해줘야 함. codex config.toml 별도 구성 필요. 훅 구성 잘 안 되면 "코덱스 훅" 검색해서 나오는 hooks+codex 페이지를 복사해 주면 알아서 만들어줌.

검증 루프(Plan·Execute·Verify) + 하네스 해부 — /harness → phase·step 분해 → executor.py
하네스 해부: /harness 스킬이 실행 진입점. 스펙을 작업 하나하나로 분해 — 페이즈(phase)와 스텝(step)별로 분해해 설계 시작. executor.py = 자율 실행 루프(에이전트 루프 스크립트). 여기서 Codex 에이전트들을 순차적으로 스핀해 돌리고, 가드레일(훅)로 품질·안전 검사. 페이즈 단위로 계속 돌아감.
좋은 스텝 원칙: (1) 자기 완결성 — 다른 것에 dependent하면 안 됨. (2) 한 스텝 = 한 레이어/한 모듈. (3) 시그니처 수준. (4) AC(Acceptance Criteria)가 매우 중요 — 실행 가능한 커맨드로 검증할 수 있어야 함. executor.py 자율 루프: 순차 선택 → 구현 → 검증 → 상태 업데이트. 상태는 4가지로 관리.
package.json 없으면 TDD 스크립트가 건너뛰도록 방어 필요(캐치 못한 부분 반영해 업데이트). 코드 리뷰는 하네스로 코드 많이 작성 시작하면 그때부터. (다음 챕터에서 AI 코드 리뷰를 하네스에 연결하는 법 다룸 — Part 2에서.)

[데모] 3시간 반 자율 실행 — 저녁 먹는 동안 5개 phase를 혼자 완성
하네스 실행 → codex가 스펙 전부 읽고 일 시작. 약 3시간 반 동안 자율 실행. 발표자는 저녁 먹으러 나갔고, 그 동안 폰으로 에이전트가 "다 됐다, 다음으로 넘어갈까?" 확인. 지시는 거의 안 함("이거 해 저거 해", 가끔 "이거 맞나요?"에 "맞아" 정도). 페이즈 0부터 시작해 총 5개:
- Phase 0 Foundation: 프로젝트 셋업, 코어 타입 생성, DB 마이그레이션 SQL 스크립트 미리 작성(나중에 Supabase MCP로 실행), 디자인 토큰(CSS)·폰트스택 인프라.
- Phase(Core): TDD로 드디어 로직 작성 시작. money normalize(금액 정규화 연산 유틸), CSV 파싱 유틸, 마스킹/해시 유틸(Claude API 보낼 때), analysis(규칙·통계 분석 — AI에 전부 의존하지 않고 코드 레벨에서 분석해 AI 비용↓·결정론적↑), orchestration(오케스트레이터).
- Phase(Dashboard/API): Claude 어댑터(AI 인사이트 로직), Supabase 어댑터(검증·조회), auth flow(Google 인증 로그인/로그아웃), analyze(얇은 와이어링/라우팅 로직), UI kit(money·percent 포맷, 테스트 인프라), 대시보드 페이지 UI.
- Phase(Polar 결제): 결제 어댑터, 체크아웃 라우트(결제 어댑터 연결), 웹훅(결제 이벤트 수신·처리), 업그레이드 UI(시각 설계 규칙 파악, 잠금 미리보기에서 프로 플랜 업그레이드 CTA).
- Phase(Landing): 랜딩 섹션 컴포넌트, 샘플 데모 섹션(키 없이 작동하는 데모 — 외부 API/인증/DB 없이).
아직 인증·결제·DB 연동은 안 함. UI와 로직만 작성하도록 시킴(PRD 스펙 그대로). 다음 섹션에서 UI 테스트/직접 보며 디버깅 예정.
